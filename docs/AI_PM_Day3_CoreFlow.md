# Day 3 作业：AI 产品全链路核心交互流程设计

> 产品：近代史纲要 · AI 刷题台  
> 主题：Vibe Coding 工具与 AI Agent 协作 —— 从用户进入产品到完成一次完整复习闭环，AI 在每个节点扮演什么角色、输入输出什么、人与 AI 的边界在哪里。

---

## 1. 设计范围

本流程覆盖一名学生从「打开网页」到「完成针对性复习并离开」的完整单次旅程，以及错题数据如何反哺下一次会话。

- **起点**：学生通过链接或二维码进入登录页。
- **终点**：学生完成刷题、查看 AI 解析、浏览错题画像，并带着薄弱点离开或继续练习。
- **闭环**：答题记录、错题标签、薄弱知识点进入数据库，用于下次登录时的个性化推荐。

---

## 2. 用户角色与状态

| 角色 | 描述 | 关键状态 |
|------|------|----------|
| 学生 | 修读《中国近现代史纲要》的本科生 | 登录态、当前题号、已答/未答、正确率 |
| AI 助手 | 背后的 `mimo-v2.5-pro-ultraspeed` 模型 | 当前题目上下文、章节知识库、历史错题 |
| 系统 | Next.js + SQLite 全栈服务 | 用户答题记录、章节正确率、错题画像 |

---

## 3. 全链路核心交互流程

```text
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  1. 学号登录  │ → │ 2. 选择模式  │ → │  3. 开始刷题  │ → │ 4. 答题反馈  │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
                                                                  │
    ┌─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 5. AI 解析  │ → │ 6. 继续/结束  │ → │ 7. 错题画像  │ → │ 8. 针对性复习 │
│  （可选）    │    │   本次练习   │    │  （可选）    │    │  （闭环）    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

## 4. 分阶段详细设计

### 阶段 1：学号登录

| 维度 | 说明 |
|------|------|
| **用户目标** | 快速进入自己的刷题空间，不希望注册密码。 |
| **界面元素** | 输入框（8 位学号）、登录按钮、产品一句话介绍。 |
| **系统行为** | 校验学号为 8 位数字；若不存在则自动创建用户；写入 iron-session。 |
| **AI 作用** | 不参与。 |
| **数据变化** | `users` 表新增或确认记录；`session` 建立。 |
| **异常处理** | 输入非 8 位数字：前端即时提示；服务端再次校验。 |

### 阶段 2：选择模式

| 维度 | 说明 |
|------|------|
| **用户目标** | 根据当前复习需求选择练习范围。 |
| **界面元素** | 模式卡片（单元测试 / 全书测试 / 错题本）、章节课本选择、题量选择（10/20/50/全部）、乱序开关。 |
| **系统行为** | 从 `questions.json` / `chapters.json` 构建题目列表；从 `answers` 表读取该用户历史作答，标记错题。 |
| **AI 作用** | 不参与。 |
| **数据变化** | 本地状态 `list` 生成；URL 参数同步，便于分享与返回。 |
| **异常处理** | 错题本为空时提示「暂无错题，先去刷题吧」。 |

### 阶段 3：开始刷题

| 维度 | 说明 |
|------|------|
| **用户目标** | 高效完成题目，获得即时反馈。 |
| **界面元素** | 题干、选项 A/B/C/D、进度条、题号导航、返回画像入口、右侧资料抽屉。 |
| **系统行为** | 记录用户选择；点击选项后立即提交；对比正确答案；更新 UI。 |
| **AI 作用** | 不参与直接答题，但右侧 AI 复习助手已可调用。 |
| **数据变化** | 写入 `answers` 表（用户答案、是否正确、时间戳）。 |
| **异常处理** | 网络失败时本地保留重试；重复作答以最新为准。 |

### 阶段 4：答题反馈

| 维度 | 说明 |
|------|------|
| **用户目标** | 知道自己错在哪里，理解正确答案。 |
| **界面元素** | 正确/错误标识、正确答案、内置解析、「AI 提示」「AI 详细解析」「AI 错题复盘」按钮。 |
| **系统行为** | 读取内置解析展示；调用 AI 时携带题目、章节、正确选项、用户错选。 |
| **AI 作用** | **生成内容**：根据章节知识库和题目，生成提示/逐项解析/错题复盘。 |
| **数据变化** | 若答错，写入 `mistakes` 表；`userStats` 视图更新。 |
| **异常处理** | AI 流超时（30s）或中断：展示「AI 解析生成失败，请重试」并保留重新请求入口。 |

### 阶段 5：AI 解析（可选但核心）

| 维度 | 说明 |
|------|------|
| **用户目标** | 不只是知道答案，而是理解每个选项背后的史实。 |
| **界面元素** | 流式输出区域、Markdown 渲染、复制/重试按钮。 |
| **系统行为** | 调用 `/api/ai/explain`，SSE 流式返回 JSON 编码内容。 |
| **AI 作用** | **解释与教学**：对 A/B/C/D 逐项说明史实背景与对错原因；错题复盘时指出用户常混淆的知识点。 |
| **数据输入** | 系统 Prompt（章节知识库 + 时间轴 + 输出格式要求）+ User Prompt（题目 + 用户答案）。 |
| **数据输出** | 结构化自然语言解析，支持 Markdown。 |
| **异常处理** | 流式输出中断：前端尝试自动重连一次；仍失败则提示用户手动重试。 |

### 阶段 6：继续 / 结束本次练习

| 维度 | 说明 |
|------|------|
| **用户目标** | 完成当前题目后决定下一步。 |
| **界面元素** | 「下一题」「查看资料」「结束并查看画像」按钮。 |
| **系统行为** | 更新本地进度；保存 `sessionStorage` 状态。 |
| **AI 作用** | 不参与。 |
| **数据变化** | 答题记录已实时写入数据库；本地状态用于返回定位。 |
| **异常处理** | 离开页面时自动保存当前题号，下次可从断点继续（由 URL / sessionStorage 恢复）。 |

### 阶段 7：错题画像

| 维度 | 说明 |
|------|------|
| **用户目标** | 了解自己的薄弱章节和知识点，制定复习计划。 |
| **界面元素** | 章节正确率柱状图、薄弱知识点 TOP 列表、高频错题 TOP 列表、返回刷题按钮。 |
| **系统行为** | 查询 `answers` / `mistakes` 聚合数据；按章节和知识点分组统计。 |
| **AI 作用** | **分析与推荐**：未来可基于画像生成「复习建议」（如「第 3 章条约类题目错误率高，建议重点复习」）。当前版本以统计可视化为主。 |
| **数据变化** | 无写入，纯读取聚合。 |
| **异常处理** | 新用户无数据时展示空状态引导。 |

### 阶段 8：针对性复习（闭环）

| 维度 | 说明 |
|------|------|
| **用户目标** | 回到具体题目或章节，把薄弱点补上。 |
| **界面元素** | 从画像点击错题 → 返回刷题页并定位原题；右侧 AI 复习助手抽屉。 |
| **系统行为** | 携带 `returnQuestionChapter`、`returnQuestionIndex`、`returnCount`、`returnShuffle` 返回刷题页；恢复题量和乱序设置。 |
| **AI 作用** | **陪伴问答**：AI 复习助手 `/api/ai/chat` 接收用户围绕当前题目或章节的自由提问，结合上下文回答。 |
| **数据变化** | 新一轮答题继续写入 `answers`，更新画像。 |
| **异常处理** | 返回参数无效时降级为默认刷题模式。 |

## 5. 人机协作边界

| 能力 | 人类（学生） | AI | 系统 |
|------|-------------|-----|------|
| 判断答案对错 | — | 不参与 | ✅ 基于标准答案判定 |
| 生成解析内容 | 阅读、评价 | ✅ 生成提示/逐项解析/复盘 | 调用 API、渲染输出 |
| 决定复习策略 | ✅ 选择模式、题量 | 提供数据建议 | 记录选择、计算画像 |
| 自由提问答疑 | ✅ 提出真实疑问 | ✅ 结合章节知识回答 | 维护上下文、限流校验 |
| 数据持久化 | — | — | ✅ SQLite 记录所有作答 |

**边界原则**：
1. AI 不替代标准答案判定，避免「幻觉改答案」。
2. AI 解析是可选增强，内置解析始终可用，保证基础体验不依赖 AI。
3. 用户可随时关闭 AI 抽屉，自主控制 AI 介入深度。

## 6. 关键数据流

### 6.1 AI 解析流

```text
用户点击「AI 详细解析」
  ↓
前端携带 {题目, 用户答案, 章节, 模式} 请求 /api/ai/explain
  ↓
服务端组装 System Prompt（章节知识库 + 输出格式）+ User Prompt
  ↓
调用 mimo-v2.5-pro-ultraspeed（stream=true, 无 max_tokens）
  ↓
SSE JSON chunk 流式返回前端
  ↓
react-markdown + remark-gfm 渲染
```

### 6.2 错题画像流

```text
用户进入 /profile
  ↓
服务端查询该 userId 的 answers / mistakes
  ↓
按 chapter 聚合正确率
  ↓
按知识点标签统计错误频次
  ↓
返回高频错题列表（合并 questions.json 题目文本）
  ↓
前端渲染图表与列表
```

### 6.3 状态恢复流

```text
用户从画像点击「返回刷题」
  ↓
URL 携带 returnQuestionChapter / returnQuestionIndex / returnCount / returnShuffle
  ↓
QuizClient 优先读取 URL 参数定位原题
  ↓
若 URL 无参数，则读取 sessionStorage 恢复
  ↓
重建题目列表并滚动到对应位置
```

## 7. 异常与兜底策略

| 异常场景 | 兜底策略 |
|----------|----------|
| AI 接口超时/失败 | 保留内置解析；提示「AI 解析暂不可用，请重试」；提供重新请求按钮。 |
| 新用户无错题画像 | 展示空状态与引导：「完成 5 道题后即可查看错题画像」。 |
| 返回参数失效 | 降级到默认全书测试模式，避免白屏。 |
| 网络波动导致提交失败 | 前端保留选择状态，失败时允许重新提交；避免重复计数以最新为准。 |
| AI 生成内容有 AI 味 | 后续迭代引入 `humanize-zh` skill，对 AI 解析示例和资料文案进行去痕润色。 |

## 9. 关键成功指标

| 指标 | 目标 | 说明 |
|------|------|------|
| 登录完成率 | > 90% | 学号登录无门槛，重点看输入校验是否友好。 |
| AI 解析使用率 | > 40% | 答错题后愿意点开 AI 解析。 |
| 错题画像访问率 | > 30% | 刷题后会主动查看个人薄弱点。 |
| 从画像返回刷题率 | > 25% | 画像不是终点，能带动二次练习。 |
| AI 解析满意度 | > 4.0/5 | 通过后续用户访谈或「有帮助/无帮助」按钮收集。 |

## 10. 可进一步迭代的 AI 协作点

1. **AI 生成复习计划**：根据画像数据，自动生成「今日复习 3 道第 3 章条约题 + 2 道第 5 章人物题」的推荐列表。
2. **AI 出题**：针对高频错题知识点，由 AI 生成变式题，扩充题库。
3. **AI 资料润色**：使用 `humanize-zh` skill 对 AI 解析示例、课程资料文案进行去痕，让学生读起来更像真人讲解。
4. **多轮追问**：在 AI 解析后支持「为什么 A 不对」「再举一例」等上下文追问。
